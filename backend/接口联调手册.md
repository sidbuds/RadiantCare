# 熙心健康体检平台后端接口联调手册

## 1. 环境准备
1. 导入建表脚本：`D:\projects\熙心健康体检平台\sql\xixin_health.sql`
2. 导入演示数据脚本：`D:\projects\熙心健康体检平台\backend\sql\xixin_health_demo.sql`
3. 启动服务：

```powershell
D:\tools\apache-maven-3.9.11\bin\mvn.cmd spring-boot:run
```

4. 默认地址：`http://127.0.0.1:8080`

## 2. 演示账号
- 用户：`user / 123456`
- 医生：`doctor / 123456`
- 运营：`operator / 123456`
- 管理员：`admin / 123456`

## 3. 用户主链路

### 3.1 用户登录
`POST /api/auth/login`

```json
{
  "username": "user",
  "password": "123456"
}
```

保存：`userToken`

### 3.2 查询可预约时段
`GET /api/appointments/available-slots?centerCode=C001&packageId=1001&date=2026-06-09`

请求头：
- `Authorization: Bearer {userToken}`

### 3.3 创建预约
`POST /api/appointments`

```json
{
  "packageId": 1001,
  "centerCode": "C001",
  "appointDate": "2026-06-09",
  "timeSlotCode": "AM01",
  "remark": "first appointment"
}
```

保存：`appointmentNo`

### 3.4 创建订单
`POST /api/orders`

```json
{
  "appointmentNo": "appointmentNo",
  "couponCode": "",
  "extraItemCodes": []
}
```

保存：`orderNo`

### 3.5 模拟支付
`POST /api/orders/{orderNo}/pay`

### 3.6 管理员登录
`POST /api/auth/login`

```json
{
  "username": "admin",
  "password": "123456"
```

## 4. 第二份报告与对比链路

### 4.1 创建第二份预约
第二份预约请改用 `2026-06-10`

```json
{
  "packageId": 1001,
  "centerCode": "C001",
  "appointDate": "2026-06-10",
  "timeSlotCode": "AM01",
  "remark": "second appointment"
}
```

### 4.2 第二份血检结果示例
```json
{
  "itemCode": "BLOOD",
  "resultEntries": [
    {
      "metricCode": "GLU",

      "resultValue": "6.2",
```

`POST /api/report-compare/tasks`

## 6. 医生分析链路

### 6.1 医生查看异常总览
- `GET /api/doctor/analytics/abnormal-overview`
- `GET /api/doctor/analytics/abnormal-distribution`
- `GET /api/doctor/analytics/high-risk-users`
- `GET /api/doctor/analytics/abnormal-trend`

推荐示例：

`GET /api/doctor/analytics/abnormal-overview?startDate=2026-06-01&endDate=2026-06-30`

成功响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "doctorId": 2,
    "startDate": "2026-06-01",
    "endDate": "2026-06-30",
    "totalResultCount": 2,
    "abnormalCount": 1,
    "abnormalRate": 0.5000,
    "highRiskCount": 1,
    "highRiskRate": 0.5000
  }
}
```

`GET /api/doctor/analytics/abnormal-distribution?startDate=2026-06-01&endDate=2026-06-30`

成功响应关键字段：
- `itemCode`
- `itemName`
- `totalCount`
- `abnormalCount`
- `highRiskCount`
- `level1Count / level2Count / level3Count`

`GET /api/doctor/analytics/high-risk-users?startDate=2026-06-01&endDate=2026-06-30`

成功响应关键字段：
- `userId`
- `userName`
- `mobile`
- `highRiskItemCount`
- `maxAbnormalLevel`
- `items`

`GET /api/doctor/analytics/abnormal-trend?startDate=2026-06-01&endDate=2026-06-30`

成功响应关键字段：
- `date`
- `totalCount`
- `abnormalCount`
- `highRiskCount`
- `abnormalRate`
- `momDelta`

数据库检查点：
- `exam_result.entry_doctor_id = 当前医生ID`
- `exam_result.entry_time` 落在查询时间范围内
- `exam_result.is_abnormal / abnormal_level` 已正确写入
- 如需按科室关联分析，需有对应 `exam_task_item` 记录

### 6.2 医生查看工作量统计
- `GET /api/doctor/analytics/workload-overview`
- `GET /api/doctor/analytics/workload-trend`
- `GET /api/doctor/analytics/workload-breakdown`

推荐示例：

`GET /api/doctor/analytics/workload-overview?startDate=2026-06-01&endDate=2026-06-30`

成功响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "doctorId": 2,
    "startDate": "2026-06-01",
    "endDate": "2026-06-30",
    "completedItemCount": 2,
    "resultEntryCount": 2,
    "reviewCount": 1,
    "consultationReplyCount": 1,
    "avgCompletionMinutes": 18.50
  }
}
```

`GET /api/doctor/analytics/workload-trend?startDate=2026-06-01&endDate=2026-06-30`

成功响应关键字段：
- `date`
- `completedItemCount`
- `resultEntryCount`
- `reviewCount`
- `consultationReplyCount`

`GET /api/doctor/analytics/workload-breakdown?startDate=2026-06-01&endDate=2026-06-30`

成功响应关键字段：
- `departmentCode`
- `departmentName`
- `itemCode`
- `itemName`
- `taskItemCount`
- `completedItemCount`
- `avgCompletionMinutes`

数据库检查点：
- `exam_task_item.item_status = 2` 会计入完成数
- `exam_task_item.start_time / complete_time` 会计入平均处理时长
- `doctor_review_record.reviewer_id = 当前医生ID` 会计入审核数
- `doctor_consultation_reply.reply_role = DOCTOR` 且 `reply_user_id = 当前医生ID` 会计入咨询回复数

### 6.3 管理员查看医生总览
- `GET /api/admin/doctor-analytics/abnormal-overview`
- `GET /api/admin/doctor-analytics/abnormal-distribution`
- `GET /api/admin/doctor-analytics/high-risk-users`
- `GET /api/admin/doctor-analytics/workload-ranking`
- `GET /api/admin/doctor-analytics/department-workload`

推荐示例：

`GET /api/admin/doctor-analytics/abnormal-overview?startDate=2026-06-01&endDate=2026-06-30&doctorId=2`

扩展示例：
- `GET /api/admin/doctor-analytics/abnormal-distribution?startDate=2026-06-01&endDate=2026-06-30&departmentCode=LAB`
- `GET /api/admin/doctor-analytics/high-risk-users?startDate=2026-06-01&endDate=2026-06-30`
- `GET /api/admin/doctor-analytics/workload-ranking?startDate=2026-06-01&endDate=2026-06-30`
- `GET /api/admin/doctor-analytics/department-workload?startDate=2026-06-01&endDate=2026-06-30`

成功响应关键字段：
- 异常总览：`totalResultCount / abnormalCount / highRiskCount`
- 工作量排行：`doctorId / doctorName / completedItemCount / resultEntryCount / reviewCount / consultationReplyCount / avgCompletionMinutes`
- 科室汇总：`departmentCode / departmentName / taskItemCount / completedItemCount / resultEntryCount / abnormalCount`

权限检查点：
- 医生账号访问 `/api/admin/doctor-analytics/**` 应返回无权限
- 管理员账号可带 `doctorId`、`departmentCode` 做筛选

## 9. 常见检查点
- 如果 `operator` 无法登录，先检查 `staff_account` 和 `staff_role_rel`
- 如果用户无法登录，先检查 `user.user_no` 和 `user.password_hash`
- 如果医生分析结果为空，先检查 `exam_result.entry_time`、`entry_doctor_id`、`exam_task_item.doctor_id`
- 如果管理员看不到医生排行，先确认已有医生结果录入、审核记录或咨询回复数据
- 如果对比任务失败，先检查两份报告是否已发布且同属一个用户
