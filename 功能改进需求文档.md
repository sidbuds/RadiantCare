# 功能改进需求文档

> 本文档收敛当前系统**所有待改进功能**的详细需求，包含现状分析、接口变更、数据库改动、前后端改动点及影响范围。
> 涵盖用户端（改进一~四）、医生端（改进五~七）、管理端（改进八、十一）和运营端（改进九~十）共 11 项改进。
# 第一部分：用户端改进

## 改进一：用户个人信息页

### 1.1 现状分析
- `user` 表已有字段：`name`、`gender`、`birthDate`、`idType`、`idNo`、`mobile`、`email`、`passwordHash`、`address`、`emergencyContact`、`emergencyMobile`
- `UserMapper` 已存在，继承 `BaseMapper<UserEntity>`，但**没有 UserService、没有 Controller 接口**
- 前端**没有个人信息页面**，`router/index.ts` 的 USER 路由下无 profile 相关路由
- 当前用户信息仅在登录时通过 `/api/auth/me` 返回 `AuthContext.LoginUser`（仅含 accountId、userId、username、displayName、role），**不返回完整个人信息**

## 改进三：咨询时共享病史与报告

### 3.1 现状分析
- 用户发起咨询时**必须关联一份已发布的报告**（`reportNo` 必填）
- 消息发送接口 `ReplyConsultationRequest` 只有 `replyContent` + `attachmentUrl`
- `attachmentUrl` 是一个可选的字符串附件链接，**没有结构化的"分享报告"或"分享病史"能力**
- 前端发消息时只有一个文本输入框，没有"分享报告"或"查看病史"的交互
- 用户无法在对话中主动发送报告给医生，无法授权医生查看健康档案

### 3.2 接口变更

#### 修改接口：POST /api/doctor-consultations/{no}/messages

新增可选字段（不破坏现有调用）：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| replyContent | String | 是 | 现有字段，不变 |
| attachmentUrl | String | 否 | 现有字段，不变 |
| sharedReportNo | String | 否 | **新增**，分享的报告编号 |
| shareHealthProfile | Boolean | 否 | **新增**，是否授权医生查看病史 |

**后端逻辑变更：**
- 若 `sharedReportNo` 不为空，校验报告属于当前用户且已发布，将报告信息以结构化数据写入消息
- 若 `shareHealthProfile` 为 true，在咨询记录上标记 `healthProfileShared = 1`

#### 新增接口：GET /api/doctor/consultations/{no}/patient-health-profile

# 第二部分：医生端改进（科室分配体系）

## 改进五：医生科室数据模型

### 5.1 现状分析
当前 `staff_account` 表**没有科室字段**：
```
id, username, password_hash, display_name, bind_user_id, status, last_login_at, ...
```
所有医生账号没有任何科室标识，系统无法区分哪个医生属于哪个科室。

演示数据中只有 1 个医生账号（doctor），没有科室信息。

### 5.2 数据库改动

#### staff_account 表新增字段
```sql
ALTER TABLE staff_account
  ADD COLUMN department_code VARCHAR(32) DEFAULT NULL COMMENT '所属科室编码',
  ADD COLUMN department_name VARCHAR(64) DEFAULT NULL COMMENT '所属科室名称',
  ADD COLUMN specialty VARCHAR(128) DEFAULT NULL COMMENT '专业方向/擅长描述',
  ADD COLUMN center_code VARCHAR(32) DEFAULT NULL COMMENT '所属体检中心编码';
```

#### 新增 doctor_department_rel 表
支持一个医生属于多个科室（如某医生同时负责内科和外科）：
```sql
CREATE TABLE doctor_department_rel (
  id BIGINT NOT NULL AUTO_INCREMENT,
  doctor_id BIGINT NOT NULL COMMENT 'staff_account.id',
  department_code VARCHAR(32) NOT NULL COMMENT '科室编码',
  department_name VARCHAR(64) DEFAULT NULL COMMENT '科室名称',
  is_primary TINYINT NOT NULL DEFAULT 0 COMMENT '是否主科室 0否 1是',
  center_code VARCHAR(32) DEFAULT NULL COMMENT '所属中心',
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  created_by BIGINT DEFAULT NULL,
  updated_by BIGINT DEFAULT NULL,
  is_deleted TINYINT NOT NULL DEFAULT 0,
  PRIMARY KEY (id),
  KEY idx_doctor (doctor_id),
  KEY idx_department (department_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 字段关系
```
staff_account.department_code  ←→  exam_department_route.department_code
doctor_department_rel.department_code  ←→  exam_department_route.department_code
```

## 改进六：体检任务按科室分配医生

### 6.1 现状分析
`DoctorAssignmentService.assignDoctor(departmentCode, centerCode, excludedDoctorIds)` 接收了科室参数，但**完全没有用它过滤医生**：
```java
// 当前代码：查询所有 status=1 的 staff_account，不按科室过滤
List<StaffAccountEntity> activeStaff = staffAccountMapper.selectList(
    new LambdaQueryWrapper<StaffAccountEntity>()
        .eq(StaffAccountEntity::getStatus, 1)
        .eq(StaffAccountEntity::getIsDeleted, 0));
// departmentCode 参数传入后完全未使用！
```
结果：检验科的检查项可能被分配给影像科医生。

### 6.2 改进后分配逻辑
```
1. 根据 departmentCode 从 doctor_department_rel 查出该科室的医生列表
2. 过滤掉 excludedDoctorIds（同一任务内避免重复分配）
3. 在剩余医生中，按待办工作量（itemStatus=0,1 的 task item 数量）升序选择
4. 如果该科室没有可用医生，降级到按 staff_account.department_code 匹配
5. 如果仍然没有，从所有医生中选工作量最小的（兜底 + 告警日志）
```

### 6.3 数据库改动
无额外改动，依赖改进五的表结构。

### 6.4 后端改动
| 文件 | 改动 | 说明 |
|---|---|---|
| `DoctorAssignmentService.java` | **核心修改** | assignDoctor() 增加按科室过滤逻辑 |

`ExamService.java` **无需改动**，现有代码已正确从 `exam_department_route` 获取 `departmentCode` 并传给 `assignFromRoute()`。

### 6.5 前端改动
**无**。体检任务生成是管理员触发的后端自动流程。

### 6.6 影响范围
- 仅修改 `DoctorAssignmentService` 内部逻辑，对外接口不变
- 不影响前端任何页面

## 改进七：咨询按科室分配医生

### 7.1 现状分析
- `selectBestDoctor()` 通过硬编码关键词匹配选医生（heart→心内科、sugar→内分泌等），不可靠
- 没有利用报告中的科室/检查项信息
- 没有结构化的科室字段，全靠 `username`/`displayName` 文本猜测
- 用户无法选择或了解被分配的医生属于什么科室

### 7.2 改进后分配逻辑
```
1. 根据用户的报告（reportNo），查出该报告关联的检查项和异常指标
2. 根据异常指标的主要科室倾向，确定目标科室
   - 例如：血常规异常 → 检验科/内科；影像异常 → 影像科
3. 从目标科室的医生中，按工作量最小选择
4. 如果报告无法确定科室，使用 consultationType 字段做关键词匹配（保留现有逻辑作为兜底）
5. 将分配结果（doctorId、departmentCode）写入 doctor_consultation
```

## 改进九：排班管理增加科室维度

### 9.1 现状分析
- `resource_capacity` 表用于排班，字段：`centerCode`、`appointDate`、`timeSlotCode`、`resourceType`、`resourceCode`、`capacityTotal`、`capacityUsed`、`capacityLocked`、`status`

## 改进十一：管理端职责重新定义

## 11.1 现状分析：当前管理员做了什么

当前 `/api/admin/*` 下的接口：

| 接口 | 当前归属 | 实际应该归属 |
|---|---|---|
| `POST /api/admin/exam-tasks/generate` — 生成体检任务 | ADMIN | **→ 运营端 OPERATOR** |
| `POST /api/admin/reports/generate` — 生成报告 | ADMIN | **→ 医生端 DOCTOR**（或保持 ADMIN，属于流程驱动） |
| `POST /api/admin/reports/{no}/review` — 审核报告 | ADMIN | **→ 医生端 DOCTOR**（医生已有审核接口） |
| `POST /api/admin/reports/{no}/publish` — 发布报告 | ADMIN | **→ 医生端 DOCTOR** |
| `POST /api/admin/consultations/{no}/assign` — 分配咨询 | ADMIN | **→ 运营端 OPERATOR**（或改为自动分配） |
| `GET /api/admin/doctor-analytics/*` — 医生分析 | ADMIN | **→ 运营端 OPERATOR** |
| `GET/POST /api/admin/refunds/*` — 退款审核 | ADMIN | **→ 运营端 OPERATOR**（运营已有退款接口） |

**问题：** 当前管理员承担了大量**业务操作**（生成任务、审核报告、分配咨询、审核退款），这些本质上是运营和医生的职责。真正的系统管理员应该聚焦于**账号权限管理和系统配置**。

## 11.2 管理员新职责定义

| 职责 | 说明 |
|---|---|
| 账号管理 | 创建/编辑/停用/启用所有角色的账号（用户、医生、运营、管理员） |
| 角色权限管理 | 为账号分配角色，管理角色-菜单-按钮权限映射 |
| 医生科室管理 | 维护医生的科室归属和专业方向（已在功能改进需求文档-改进八中定义） |
| 数据字典管理 | 维护系统字典（科室编码、状态码、检查项分类等） |
| 系统参数管理 | 维护全局配置参数（预约规则、分享有效期、密码策略等） |
| 审计日志查看 | 查看关键操作的审计记录（审核、退款、权限变更等） |
