# 功能改进实施计划

> 基于 11 项改进的依赖关系分析，划分为 5 个阶段。
> 每个阶段内任务互不依赖、可并行开发，后续阶段不会修改前序阶段已交付的文件。

---

## 依赖关系分析

```
改进五（科室数据模型）
  ├── 改进六（体检按科室分配）    依赖五的表结构
  ├── 改进七（咨询按科室分配）    依赖五的表结构
  ├── 改进八（管理员医生管理）    依赖五的表结构
  └── 改进九（排班科室化）        依赖五的科室体系

改进二（健康档案）
  └── 改进三（咨询共享病史）      依赖二的数据

改进四（引导实时化）             无依赖，独立修复
改进一（个人信息页）             无依赖，独立功能
改进十（运营分析科室维度）       依赖九的排班科室化
改进十一（管理端职责重定义）     依赖八（医生管理已建好）
```

## 阶段一：基础设施 + 独立修复

**目标：** 建立科室数据模型，修复现有 bug，补齐基础用户功能
**无任何前置依赖，可立即开始**
**修改文件范围：** StaffAccountEntity、DoctorDepartmentRelEntity/Mapper、ExamService、ExamTaskGuide.vue、UserProfile 相关新文件

### 任务 1.1：改进五 — 医生科室数据模型
- `staff_account` 表新增 4 个字段（department_code、department_name、specialty、center_code）
- 新建 `doctor_department_rel` 表（含唯一约束 `uk_doctor_department (doctor_id, department_code, is_deleted)`）
- 新增 `DoctorDepartmentRelEntity.java` + `DoctorDepartmentRelMapper.java`
- `StaffAccountEntity.java` 新增对应字段
- 补充索引和演示数据（多科室医生账号）

### 任务 1.2：改进四 — 体检任务引导实时化（Bug 修复）
- `ExamService.guide()` 方法重写：关联查询 `exam_department_route` 表
- 新增 `exam/dto/GuideItemVO.java` 合并 task item + route 引导字段
- 前端 `ExamTaskGuide.vue` 增加状态视觉区分（已完成/进行中/待检）
- 验证引导数据完整显示（guideText、needEmptyStomach、buildingNo、floorNo）

### 任务 1.3：改进一 — 用户个人信息页
- 新增 `user/service/UserProfileService.java` + `user/controller/UserProfileController.java`
- 新增 `user/dto/UpdateProfileRequest.java` + `user/dto/ChangePasswordRequest.java`
- 新增 `views/user/UserProfile.vue`
- `router/index.ts` 增加 USER 路由 `{ path: 'profile', ... }`
- 新增 `api/user.ts`（getProfile、updateProfile、changePassword）

**交付物：** 科室模型可用，引导页 bug 修复，用户可管理个人信息

## 阶段二：科室化核心链路

**目标：** 让医生分配和排班真正按科室运转
**前置：** 阶段一完成（科室数据模型就绪）
**修改文件范围：** DoctorAssignmentService、ResourceCapacityEntity、SaveScheduleRequest、OperatorScheduleService/Controller、ScheduleManage.vue、HealthProfile 相关新文件
**不触碰阶段一文件：** 不改 StaffAccountEntity、ExamService、UserProfile 相关文件

### 任务 2.1：改进六 — 体检任务按科室分配医生
- `DoctorAssignmentService.assignDoctor()` 重写内部逻辑：
  1. 先查 `doctor_department_rel` 按 departmentCode 过滤
  2. 按待办工作量升序选择
  3. 兜底：按 `staff_account.department_code` 匹配 → 全部医生中选
- 接口签名不变，仅改内部逻辑
- 更新演示数据确保科室编码一致
- 验证：生成任务后，检验科检查项分配给检验科医生

### 任务 2.2：改进九 — 排班管理增加科室维度
- `resource_capacity` 表新增 `department_code`、`department_name`
- `ResourceCapacityEntity.java` 新增字段
- `SaveScheduleRequest.java` 新增 `departmentCode`、`departmentName`
- `OperatorScheduleService.java`：list() 增加 departmentCode 过滤；create()/update() 写入科室
- `OperatorScheduleController.java`：list() 增加 departmentCode 查询参数
- 前端 `ScheduleManage.vue`：筛选区增加科室下拉框；表单增加科室选择；表格增加科室列
- `api/operator.ts`：ScheduleFormData 和 getOperatorSchedules 参数扩展

### 任务 2.3：改进二 — 健康档案页面
- 新增 `user/service/HealthProfileService.java` + `user/controller/HealthProfileController.java`
- 新增 `user/dto/SaveHealthProfileRequest.java`
- 新增 `views/user/HealthProfile.vue`
- `router/index.ts` 增加 USER 路由 `{ path: 'health-profile', ... }`
- `api/user.ts` 增加 getHealthProfile、saveHealthProfile

**交付物：** 体检任务按科室正确分配，排班支持按科室管理，用户可维护健康档案

## 阶段三：咨询体系重构

**目标：** 重构咨询分配逻辑 + 咨询中共享能力，作为一个完整任务执行
**前置：** 阶段一（科室模型）+ 阶段二（健康档案就绪）
**修改文件范围：** ConsultationService、CreateConsultationRequest、DoctorConsultationEntity、DoctorConsultationReplyEntity、ReplyConsultationRequest、ConsultationController、DoctorConsultationController、PublicController、ConsultDoctor.vue、ConsultationTodo.vue、consultation.ts
**不触碰阶段一二文件**

### 任务 3.1：改进七 + 改进三 — 咨询体系重构（合并执行）

#### 子任务 A：咨询按科室分配医生（改进七）
- `ConsultationService.selectBestDoctor()` 重写：
  1. 新增 `determineTargetDepartment(report)` 方法，从报告异常指标确定目标科室
  2. 从目标科室医生中按工作量选择
  3. 保留关键词匹配作为兜底
- `CreateConsultationRequest.java` 新增 `preferredDepartmentCode`
- `DoctorConsultationEntity.java` 新增 `departmentCode`、`departmentName`
- `doctor_consultation` 表新增 `department_code`、`department_name`
- `PublicController.java` 新增 `GET /api/public/departments`
- 前端 `ConsultDoctor.vue` 增加科室选择下拉框
- 新增 `api/department.ts`

#### 子任务 B：咨询时共享病史与报告（改进三）
- `ReplyConsultationRequest.java` 新增 `sharedReportNo`、`shareHealthProfile`
- `CreateConsultationRequest.java` 新增 `shareHealthProfile`（与子任务 A 合并处理该 DTO）
- `DoctorConsultationReplyEntity.java` 新增 `messageType`、`refReportNo`
- `doctor_consultation` 表新增 `health_profile_shared`（与子任务 A 合并执行 DDL）
- `doctor_consultation_reply` 表新增 `message_type`、`ref_report_no`
- `ConsultationService.java` 的 `create()` 增加病史授权逻辑；`userMessage()` 增加分享报告/病史逻辑
- `ConsultationController.java` 或 `DoctorConsultationController.java` 新增 `GET /{no}/patient-health-profile`
- 前端 `ConsultDoctor.vue` 增加分享报告按钮和分享病史开关（与子任务 A 合并改同一页面）
- 前端 `ConsultationTodo.vue` 适配新消息类型（REPORT_SHARE/PROFILE_SHARE 显示卡片）+ 查看病史按钮
- `api/consultation.ts` 参数扩展 + 新增 getPatientHealthProfile

**执行顺序：** 子任务 A 先完成核心分配逻辑，子任务 B 再在同一文件上追加共享能力。
**交付物：** 咨询按科室分配，用户可在对话中分享报告和授权病史

## 阶段四：管理端重构 + 医生管理

**目标：** 重建管理员后台，迁移现有 admin 接口到正确角色
**前置：** 阶段一（科室模型，用于医生管理）
**不触碰阶段一二三文件**

### 任务 4.1：改进八 — 管理员维护医生科室
- 新增 `admin/service/AdminDoctorService.java` + `admin/controller/AdminDoctorController.java`
- 新增 `admin/dto/UpdateDoctorRequest.java` + `admin/dto/AddDoctorDepartmentRequest.java`
- 新增 `views/admin/DoctorManage.vue`
- `router/index.ts` 增加管理员医生管理路由
- `api/admin.ts` 增加医生管理接口

### 任务 4.2A：核心业务接口迁移（体检、报告、退款）
- 体检任务生成：`AdminExamController` 标记 @Deprecated，新建 `OperatorExamController`（路径 `/api/operator/exam-tasks/generate`）
- 报告生成/发布：`AdminReportController` 标记 @Deprecated，`DoctorReportController` 增加 `generate()`、`publish()`（权限校验：只有负责该任务的医生可操作）
- 退款审核：`AdminRefundController` 标记 @Deprecated（运营端已有 `/api/operator/refunds/*`）
- 前端：`ExamTaskGenerate.vue` 移到运营菜单；`ReportPublish.vue` 的生成/发布功能整合到 `ReportReviewTodo.vue`

### 任务 4.2B：咨询和分析接口迁移（依赖阶段三完成后执行）
- 咨询分配：`AdminConsultationController` 标记 @Deprecated，新建 `OperatorConsultationController`（路径 `/api/operator/doctor-consultations/{no}/assign`）
- 医生分析：`AdminDoctorAnalyticsController` 标记 @Deprecated，新建 `OperatorDoctorAnalyticsController`（路径 `/api/operator/doctor-analytics/*`）
- 前端：`ConsultationAssign.vue` 移到运营菜单；`DoctorAnalytics.vue` 移到运营菜单
- **说明：** 此任务在阶段三之后执行，确保 ConsultationService 的所有改动已稳定

### 任务 4.3：管理员新后台
- 用户管理：`AdminUserService` + `AdminUserController`（5 接口：列表/详情/创建/修改/启停）+ `UserManage.vue`
- 角色管理：角色列表、账号角色绑定/解绑（3 接口）+ `RoleManage.vue`
- `router/index.ts` 管理员路由全面更新为新页面
- `api/admin.ts` 增加用户管理和角色管理接口

**交付物：** 管理员可管理医生科室和用户账号，admin 业务接口已迁移到正确角色

### 任务 5.1：改进十 — 运营分析增加科室维度
