# 熙心健康体检平台全链路业务说明文档

## 1. 文档目的
本文档用于把平台所有核心角色和业务链路“像业务现场一样”跑通说明，确保在继续编码前先统一业务理解。本文档以用户提供的用例图为基础，补齐普通用户、医生/医技录入人员、运营人员、系统管理员四类角色的完整链路。

## 2. 用例图角色映射

### 2.0 游客
- 浏览平台首页
- 查看体检套餐
- 查看套餐详情与检查项说明
- 查看中心介绍与基础时段展示
- 查看体检流程、注意事项、常见问题
- 跳转注册/登录

### 2.1 普通用户
- 健康档案维护
- 查询可预约时段
- 创建订单
- 支付订单
- 申请退款
- 查看健康概览
- 查看异常指标
- 历年报告对比
- 下载/分享报告

## 3. 全局主链路
1. 用户维护健康档案。
2. 用户选择套餐、中心、时段并预约。
3. 用户创建订单并支付。
4. 平台基于订单生成体检任务和导检路线。
5. 用户到院并按路线完成多个科室检查。
6. 医生/医技录入检查结果并补充结论或附件。
7. 平台按套餐模板自动归集生成报告。
8. 医生/质控审核并发布报告。
9. 用户查看报告、异常项、历年对比和健康建议。
10. 用户基于报告发起咨询医生，医生人工回复。
11. 运营人员围绕预约、订单、套餐、排班、退款和经营分析进行管理。
12. 系统管理员围绕用户、权限、字典、配置和审计进行管理。

## 4. 游客完整链路

### 4.1 游客浏览与转化链路
1. 游客打开平台首页，先看到热门套餐、中心介绍、价格区间和平台说明。
2. 游客进入套餐详情页，查看例如“呼吸道体检 + 血检”这类套餐示例，理解它只是众多套餐中的一种。
3. 游客查看套餐包含的检查项、适用人群、检查前注意事项、体检流程说明和常见问题。
4. 游客可查看中心基础排班展示或部分公开时段信息，用于判断近期是否方便体检。
5. 当游客进一步选择中心、日期并尝试进入预约、下单、咨询等动作时，系统提示先登录或注册。
6. 游客登录后，系统尽量带上刚才浏览的套餐、中心、日期等上下文，继续进入正式用户链路。

## 5. 普通用户完整链路

### 5.1 档案准备链路
1. 用户注册/登录。
2. 完善姓名、性别、年龄、身份证、手机号、联系人等信息。
3. 维护过敏史、既往病史、家族史、用药情况。
4. 若一个账号对应多个体检人，则先选择体检人后再进入后续预约。

### 5.2 预约下单链路
1. 用户浏览套餐，查看检查项、注意事项、适用人群。
2. 选择体检中心、日期、时段，查询可预约资源。
3. 提交预约申请，系统校验容量、重复预约和规则约束。
4. 预约成功后创建订单。
5. 用户确认金额并支付。
6. 如未支付，订单停留在待支付；如已支付，进入待生成任务阶段。

### 5.3 到院执行链路
1. 用户到院后，查看本次体检任务和导检路线。
2. 导检路线明确告知先去哪个科室、楼层、房间和注意事项。
3. 用户依次完成血检、影像、超声、问诊等项目。
4. 每完成一个项目，平台同步更新任务项状态。
5. 若有漏检、复检、改期补检，平台保留对应任务状态。

### 5.4 报告查看链路
1. 所有项目完成后，平台生成结构化报告。
2. 医生审核通过后，报告正式发布。
3. 用户查看报告列表、详情、异常项和总体结论。
4. 用户下载 PDF 或分享报告给家属/医生。

### 5.5 报告后续链路
1. 用户基于报告或异常项发起咨询医生。
2. 用户查看历年报告对比，了解指标趋势。
3. 用户查看系统输出的健康建议、复查建议和风险提示。
4. 如订单需取消或变更，用户发起退款申请并跟踪进度。

## 6. 医生/医技完整链路

### 6.1 接诊与执行链路
1. 医生登录后查看待处理任务列表。
2. 按科室查看分配给自己的任务项。
3. 点击开始检查，任务项状态进入检查中。
4. 根据项目类型录入数值结果、文本结论或上传附件。
5. 填写异常等级、参考范围、处理建议。
6. 完成任务项后提交结果。

### 6.2 报告处理链路
1. 平台聚合结果生成报告草稿。
2. 医生或质控人员查看报告章节与异常汇总。
3. 对草稿报告执行审核，通过、驳回或补充说明。
4. 审核通过后发布报告。
5. 如发现问题，可撤回报告并重新修订。

### 6.3 异常与高风险处理链路
1. 医生查看异常指标明细。
2. 平台按录入医生、项目、日期、科室聚合异常数据，输出异常总量、异常率和高风险量。
3. 医生查看个人异常趋势，识别近期异常上升项目和高风险人群。
4. 管理员按医生、科室查看异常汇总，用于医疗质控和资源配置。
5. 导出异常数据用于线下复核或质控。
6. 对高风险问题建立人工处理闭环。

### 6.4 咨询回复链路
1. 医生在咨询管理中查看用户问题。
2. 结合当前报告、异常项和历年趋势人工回复。
3. 更新咨询状态为已回复或已完成。
4. 保留所有回复记录和处理时间。

### 6.5 工作量统计链路
1. 平台统计医生完成的任务项数、结果录入数、报告审核数、咨询回复数。
2. 统计检查开始到完成的平均处理时长。
3. 医生只能查看个人工作量，管理员可查看全院、科室、指定医生的汇总数据。
4. 支持按日期、科室、人员维度查看工作量。
5. 为排班优化和绩效统计提供依据。

## 7. 运营人员完整链路

### 7.1 套餐配置链路
1. 运营人员创建或维护体检套餐。
2. 配置套餐价格、检查项、适用人群、上下架状态。
3. 绑定导检规则和报告模板。
4. 分析套餐销售表现并优化套餐结构。

### 7.2 排班与预约管理链路
1. 维护体检中心排班、日期、时段、容量。
2. 维护不同中心、时段的可预约规则。
3. 在预约管理中查看各时段预约人数、取消率、爽约率。
4. 根据趋势动态调整资源配置。

### 7.3 订单与退款管理链路
1. 在订单管理中查看订单状态、支付状态、来源渠道。
2. 审核退款申请，判断是否满足退款条件。
3. 跟踪退款进度和退款结果。
4. 导出订单明细、支付明细、退款明细。

### 7.4 运营分析链路
1. 查看运营数据看板。
2. 分析预约趋势、订单转化、退款率。
3. 分析套餐销量、中心利用率、时段利用率。
4. 根据数据结果优化套餐、活动和排班策略。

## 8. 系统管理员完整链路

### 8.1 用户与权限链路
1. 创建、禁用、启用后台用户。
2. 配置角色、菜单、数据权限。
3. 确保普通用户、医生、运营、管理员权限边界清晰。

### 8.2 基础配置链路
1. 维护数据字典、状态字典、异常等级字典。
2. 维护系统参数，如预约规则、分享有效期、审计保留时长。
3. 维护通知模板和基础开关。

### 8.3 审计与安全链路
1. 查看关键操作审计日志。
2. 跟踪报告审核、发布、撤回、退款审核、权限变更等敏感操作。
3. 检查异常登录、越权访问和配置错误风险。

## 9. 典型套餐链路不是唯一链路
用户举的“呼吸道体检 + 血检”是典型链路，但平台必须支持：
- 检验类链路
- 影像类链路
- 门诊问诊类链路
- 女性专项链路
- 中老年慢病筛查链路
- 多科室混合链路
- 漏检/补检/复检链路
- 模板升级后的多版本报告链路

## 10. 编码前的统一结论
- 平台包含五类角色：游客、普通用户、医生/医技录入人员、运营人员、系统管理员。
- 平台设计必须以“多角色、多科室、多套餐、多状态、多年度数据沉淀”为前提。
- 后端不能只围绕一个套餐示例设计，而要围绕通用任务拆分、结果录入、模板聚合、角色协同来设计。
- 运营链路、医生链路、管理员链路都属于核心主链路，不是边缘功能。
- 游客承担浏览和转化入口职责，不参与个人健康数据、预约订单、报告咨询等核心医疗闭环。
- 后续数据库、API、后端代码都应以本文档为业务基准。
