# 熙心健康体检平台-需求文档

## 1. 文档目标
本文档作为项目**唯一产品需求入口**，用于统一描述熙心健康体检平台的目标、用户角色、业务链路与已落地能力，替代仓库中多份零散说明文档。

## 2. 项目目标
- 打通“预约 -> 订单 -> 到院 -> 检查 -> 报告 -> 咨询 -> 对比 -> 建议”的体检主链路。
- 支持游客浏览、用户自助、医生执行、运营管理和管理员审核的多角色协作。
- 支持多套餐、多中心、多检查项、多医生协同的体检业务，而不是单一套餐示例。
- 同时提供面向用户的服务体验与面向机构的运营、风控、可追溯能力。

## 3. 用户角色
| 角色 | 说明 |
|---|---|
| 游客 | 未登录状态下浏览套餐、中心、流程、FAQ 等公开信息 |
| 普通用户 | 注册登录后完成预约、下单、支付、查看任务、查看报告、发起咨询、历年对比 |
| 医生/医技 | 查看待办检查、录入结果、审核报告、回复咨询、查看异常与工作量分析 |
| 运营人员 | 管理套餐、排班、预约、订单、退款和运营分析 |
| 系统管理员 | 执行体检任务生成、报告生成发布、咨询分配、医生分析等系统级操作 |

## 4. 已落地功能总览
当前系统已完成主链路闭环，其中**五大角色均已有独立页面与后端接口实现**。

### 4.1 游客能力
- 浏览首页、套餐列表、套餐详情、中心列表、体检流程、常见问题。
- 调用公开 API 获取套餐、中心、时段与公开内容数据。
- 在触发预约、下单、咨询等动作时，跳转登录或注册。

### 4.2 普通用户能力
- 注册、登录、查看当前登录用户信息。
- 查看可预约时段、创建预约、查看我的预约、预约详情、取消预约。
- 基于预约创建订单、订单详情、订单支付、申请退款。
- 查看体检任务、任务详情、导检信息、任务项。
- 查看我的报告、报告详情、报告指标项。
- 发起双报告对比，查看对比结果、对比导出、健康建议。
- 发起医生咨询、查看咨询列表、查看咨询详情、发送消息、关闭咨询。

### 4.3 医生/医技能力
- 查看待检任务、任务项详情、开始检查、提交检查结果、完成检查。
- 查看报告审核待办、审核历史、报告详情、执行审核。
- 查看待回复咨询、咨询详情、医生回复。
- 查看异常总览、异常分布、高风险用户、异常趋势。
- 查看工作量总览、工作量趋势、工作量明细。

### 4.4 运营人员能力
- 套餐管理：查询、新增、修改套餐及其检查项。
- 排班管理：查询、新增、修改中心时段与容量。
- 预约管理：查询、查看、取消预约。
- 订单管理：查询、查看、导出订单。
- 退款审核：查询退款申请、查看详情、通过、驳回。
- 运营分析：查看总览、预约趋势、订单转化、套餐分析。

### 4.5 系统管理员能力
- 按订单生成体检任务。
- 按任务生成报告，审核报告，发布报告。
- 将咨询工单分配给医生。
- 查看医生维度与科室维度分析数据（异常、高风险用户、工作量、科室工作量）。
- 当前版本已具备退款审核的管理员接口能力。

## 5. 业务主链路
### 5.1 游客浏览链路
1. 游客进入首页，查看体检套餐、中心介绍、流程与 FAQ。
2. 游客查看套餐详情、中心信息与时段展示。
3. 当用户点击预约、下单或咨询时，系统引导进入登录或注册。
4. 登录成功后保留上下文，继续进入正式业务流程。

### 5.2 用户主链路
1. 用户注册或登录。
2. 浏览套餐并选择中心、日期、时段。
3. 创建预约并生成订单。
4. 支付订单。
5. 系统生成体检任务并提供导检。
6. 医生执行检查并录入结果。
7. 系统生成报告，医生审核并发布。
8. 用户查看报告、异常、对比、健康建议，必要时发起咨询。

### 5.3 医生执行链路
1. 查看待检任务。
2. 开始检查并录入结果。
3. 查看待审核报告并完成审核。
4. 回复用户咨询。
5. 通过异常分析与工作量统计进行质量跟踪。

### 5.4 运营管理链路
1. 维护套餐和排班。
2. 跟踪预约与订单。
3. 审核退款申请。
4. 通过运营看板查看预约、订单、转化和套餐表现。

### 5.5 管理员审核链路
1. 根据订单生成体检任务。
2. 根据任务生成报告。
3. 审核并发布报告。
4. 分配咨询工单。
5. 查看医生与科室维度的分析数据。

## 6. 业务边界说明
当前系统已落地“主链路可用”，但以下能力在当前仓库中仍未完整形成闭环：
- 用户健康档案维护接口未开放，当前仅有数据库表设计。
- 报告下载、分享、导出等能力尚未完整落地。
- 用户管理、角色权限管理、数据字典管理、系统配置管理、审计日志管理尚未形成管理员后台模块。
- 公开站点内容（FAQ、体检指南等）目前以后端静态配置接口为主，尚未形成内容管理后台。

因此，本需求文档以**当前系统真实可运行能力**为准，不把未来规划与已落地功能混在一起。

## 7. 验收口径
当前系统按以下口径视为已实现：
- 游客公开浏览链路可用。
- 用户预约、订单、支付、任务、报告、咨询、对比、建议链路可用。
- 医生检查执行、报告审核、咨询回复、异常与工作量分析可用。
- 运营套餐、排班、预约、订单、退款、分析看板可用。
- 管理员任务生成、报告发布、咨询分配、医生分析可用。
